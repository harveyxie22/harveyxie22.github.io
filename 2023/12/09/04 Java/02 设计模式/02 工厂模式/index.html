工厂模式

Harvey Xie 2023-12-09 15:30:00
Categories： Java Tags： 设计模式

工厂模式（Factory Pattern）是 Java 中最常用的设计模式之一，有时该模式也被称为简单工厂模式。这种类型的设计模式属于创建型模式，它提供了一种创建对象的最佳方式。在工厂模式中，我们在创建对象时不会对客户端暴露创建逻辑，并且是通过使用一个共同的接口来指向新创建的对象。

在 Java 中，有三种常见的工厂模式：

  1. 简单工厂模式：是最基本的工厂模式，它通过一个工厂类来创建不同类型的对象。客户端只需要传递一个参数给工厂类，就可以获得所需的对象。
  2. 工厂方法模式：通过定义一个工厂接口来创建对象，每个具体的工厂类都实现该接口并负责创建自己的对象。这样可以让一个类的实例化延迟到其子类中，并且封装了实例化的过程。
  3. 抽象工厂模式：是最复杂的一种工厂模式，它通过定义一个抽象工厂接口来创建一组相关或相互依赖的对象。每个具体的工厂类都实现该接口并负责创建属于自己的对象。

工厂模式

简单工厂模式

// 定义一个接口
public interface Shape {
    void draw();
}

// 实现接口的具体类
public class Circle implements Shape {
    @Override
    public void draw() {
        System.out.println("Circle.draw()");
    }
}

// 实现接口的具体类
public class Rectangle implements Shape {
    @Override
    public void draw() {
        System.out.println("Rectangle.draw()");
    }
}

// 实现工厂类
public class ShapeFactory {
    public static Shape getShape(String shapeType) {
        if (shapeType == null) {
            return null;
        }
        if (shapeType.equalsIgnoreCase("CIRCLE")) {
            return new Circle();
        } else if (shapeType.equalsIgnoreCase("RECTANGLE")) {
            return new Rectangle();
        } else {
            return null;
        }
    }
}

// 客户端调用
public class Client {
    public static void main(String[] args) {
        Shape circle = ShapeFactory.getShape("Circle");
        circle.draw();
    }
}

工厂方法模式

// 定义一个接口
public interface Shape {
    void draw();
}

// 实现接口的具体类
public class Circle implements Shape {
    @Override
    public void draw() {
        System.out.println("Circle.draw()");

// 实现接口的具体类
public class Rectangle implements Shape {
    @Override
    public void draw() {
        System.out.println("Rectangle.draw()");
    }
}

// 定义一个工厂接口
public interface ShapeFactory {
    Shape getShape();
}

// 实现工厂接口的具体类
public class CircleFactory implements ShapeFactory {
    @Override
    public Shape getShape() {
        return new Circle();
    }
}

// 实现工厂接口的具体类
public class RectangleFactory implements ShapeFactory {
    @Override
    public Shape getShape() {
        return new Rectangle();
    }
}

抽象工厂模式

// 定义形状接口
public interface Shape {
    void draw();
}

// 实现形状接口的具体类
public class Circle implements Shape {
    @Override
    public void draw() {
        System.out.println("Circle.draw()");
    }
}

// 实现形状接口的具体类
public class Rectangle implements Shape {
    @Override
    public void draw() {
        System.out.println("Rectangle.draw()");
    }
}

// 定义颜色接口
public interface Color {
    void fill();
}

// 实现颜色接口的具体类
public class Red implements Color {
    @Override
    public void fill() {
        System.out.println("Red.fill()");
    }
}

// 实现颜色接口的具体类
public class Blue implements Color {
    @Override
    public void fill() {
        System.out.println("Blue.fill()");
    }
}

// 定义抽象工厂接口
public interface AbstractFactory {
    Shape getShape(String shapeType);
    Color getColor(String colorType);
}

// 实现抽象工厂接口的具体类
public class ShapeFactory implements AbstractFactory {
    @Override
    public Shape getShape(String shapeType) {
        if (shapeType.equalsIgnoreCase("CIRCLE")) {
            return new Circle();
        } else if (shapeType.equalsIgnoreCase("RECTANGLE")) {
            return new Rectangle();
        } else {
            return null;
        }
    }

    @Override
    public Color getColor(String colorType) {
        return null;
    }
}

// 实现抽象工厂接口的具体类
public class ColorFactory implements AbstractFactory {
    @Override
    public Shape getShape(String shapeType) {
        return null;
    }

    @Override
    public Color getColor(String colorType) {
        if (colorType.equalsIgnoreCase("RED")) {
            return new Red();
        } else if (colorType.equalsIgnoreCase("BLUE")) {
            return new Blue();
        } else {
            return null;
        }
    }
}

// 客户端调用
public class Client {
    public static void main(String[] args) {
        AbstractFactory shapeFactory = new ShapeFactory();
        Shape circle = shapeFactory.getShape("Circle");
        circle.draw();

        AbstractFactory colorFactory = new ColorFactory();
        Color red = colorFactory.getColor("Red");
        red.fill();
    }
}

优缺点

特点\模式 简单工厂模式 工厂方法模式 抽象工厂模式
优点 1. 简单工厂模式的实现简单，客户端可以通过工厂类来获取所需的对象，而不需要知道具体的创建逻辑；
2. 对象的创建和使用被分离，客户端只需要知道接口和工厂类即可，降低了耦合度。		 1. 工厂方法模式符合开闭原则，当需要增加新的产品时，只需要添加新的工厂类即可，不需要修改现有的代码；
2. 每个具体的工厂类负责创建自己的对象，遵循了单一职责原则，降低了代码的耦合度。		 1. 抽象工厂模式可以创建一组相关或相互依赖的对象，适用于复杂的产品族场景。
2. 客户端可以完全抽象地使用产品，不需要知道具体的工厂和产品。
缺点 1. 随着产品类型的增多，工厂方法可能会变得比较复杂，违背了开闭原则；
2. 工厂类集中了所有产品的创建逻辑，一旦出现问题，会影响所有产品的创建。		 1. 需要为每个产品定义一个具体的工厂类，会导致类的数量增加。
2. 客户端需要知道所需产品的具体工厂类，增加了客户端与具体工厂的依赖。		 1. 当需要增加新的产品时，需要修改抽象工厂接口以及所有的具体工厂类，违背了开闭原则。
2. 增加新的产品族会导致抽象工厂接口和所有具体工厂类的修改，系统变得复杂。

经典案例

java.util.Calendar

在 Java 中，java.util.Calendar 类使用了工厂方法模式来创建和获取特定的日历实例。这个工厂方法是 Calendar.getInstance()。

该方法是一个静态方法，它根据当前的时区和区域设置返回一个特定类型的日历对象。具体返回的日历对象类型取决于不同的条件，如时区、区域设置等。

工厂方法的设计思想是将对象的创建逻辑封装在一个工厂类中，通过调用工厂方法来创建对象，而不是直接使用构造函数。这样做的好处是客户端代码不需要知道具体的创建逻辑，降低了代码的耦合度。

在 Calendar 类中，通过使用工厂方法 getInstance() 来获取日历实例，代码示例如下：

import java.util.Calendar;

public class Main {
    public static void main(String[] args) {
        Calendar calendar = Calendar.getInstance(); // 获取默认的日历对象

        // 可以根据不同的时区和区域设置获取特定的日历实例
        // 例如：
        // Calendar calendar = Calendar.getInstance(TimeZone.getTimeZone("GMT+8"));
        // Calendar calendar = Calendar.getInstance(Locale.CHINA);

        // 使用日历对象进行操作
        int year = calendar.get(Calendar.YEAR);
        int month = calendar.get(Calendar.MONTH) + 1; // 月份从 0 开始，需要加 1
        int dayOfMonth = calendar.get(Calendar.DAY_OF_MONTH);

        System.out.println("当前日期：" + year + "-" + month + "-" + dayOfMonth);
    }
}

通过调用 Calendar.getInstance() 方法，我们无需关注具体的日历对象是如何创建的，而是通过工厂方法来获取了一个特定的日历实例。这使得代码更加简洁、可读性更好，并且在不同的环境中可以根据配置灵活地返回不同类型的日历实例。

Log4j 日志框架中的 Logger 类

Log4j 日志框架中的 Logger 类使用了工厂模式来创建和获取日志记录器对象。这个工厂方法是 Logger.getLogger()。

工厂方法模式将对象的创建逻辑封装在一个工厂类中，通过调用工厂方法来创建对象，而不是直接使用构造函数。在 Log4j 中，Logger 类充当了工厂类的角色，getLogger() 方法是工厂方法，用于创建和获取日志记录器对象。

Logger.getLogger() 方法接受一个字符串参数，用于指定要创建的日志记录器对象的名称。通常，我们会使用类的全限定名作为日志记录器对象的名称，以便在日志输出中能够准确地识别和定位日志来源。

以下是使用 Log4j 的示例代码：

javaCopy Codeimport org.apache.log4j.Logger;

public class Main {
    private static final Logger logger = Logger.getLogger(Main.class);

    public static void main(String[] args) {
        // 使用日志记录器对象输出日志信息
        logger.debug("Debug message");
        logger.info("Info message");
        logger.warn("Warning message");
        logger.error("Error message");
        logger.fatal("Fatal message");
    }
}

在上述代码中，通过 Logger.getLogger(Main.class) 方法获取了一个名为 “Main” 的日志记录器对象。然后，我们可以使用该日志记录器对象调用不同级别的日志输出方法，如 debug()、info()、warn() 等，来输出对应级别的日志信息。

Log4j 会根据配置文件中的日志级别设置来决定是否输出特定级别的日志信息。通过工厂方法获取日志记录器对象，我们可以在整个应用程序中共享和复用同一个日志记录器，方便地进行统一的日志管理和控制。

Android 中的 LayoutInflater 类

Android 中的 LayoutInflater 类使用了工厂模式来动态地创建视图对象。LayoutInflater 是用于将 XML 布局文件解析为相应的视图对象的类。

在 Android 中，我们通常使用 XML 布局文件来定义应用程序的用户界面。LayoutInflater 类充当了一个工厂类的角色，inflate() 方法是工厂方法，用于根据布局文件创建视图对象。

inflate() 方法接受两个参数：要解析的布局文件的资源 ID 和可选的父视图。它会解析指定的布局文件，并将其转换为对应的视图对象。如果提供了可选的父视图参数，那么解析出来的视图对象将被添加到该父视图中。

以下是使用 LayoutInflater 的示例代码：

javaCopy Codeimport android.content.Context;
import android.view.LayoutInflater;
import android.view.View;
import android.view.ViewGroup;
import androidx.appcompat.app.AppCompatActivity;

public class MainActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_main);

        // 获取 LayoutInflater 实例
        LayoutInflater inflater = (LayoutInflater) getSystemService(Context.LAYOUT_INFLATER_SERVICE);

        // 解析布局文件并创建视图对象
        View rootView = inflater.inflate(R.layout.custom_view, (ViewGroup) findViewById(R.id.container), false);

        // 将视图对象添加到父视图中
        ViewGroup parentView = findViewById(R.id.container);
        parentView.addView(rootView);
    }
}

在上述代码中，我们首先通过 getSystemService() 方法获取系统服务 LayoutInflater 的实例。然后，通过调用 inflate() 方法，传入要解析的布局文件的资源 ID 和可选的父视图，来创建一个视图对象。

最后，我们可以将创建的视图对象添加到指定的父视图中，以在用户界面中显示出来。

通过使用 LayoutInflater 类的工厂方法 inflate()，我们可以在运行时动态地创建视图对象，使得应用程序的界面更加灵活和可扩展。
