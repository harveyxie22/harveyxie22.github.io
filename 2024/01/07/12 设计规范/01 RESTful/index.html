RESTful API

Harvey Xie 2024-01-07 15:00:00
Categories： 设计规范 Tags： RESTful

RESTful 概述

什么是 RESTful API

RESTful API （REpresentational State Transfer，简称 REST）是一种基于 HTTP 协议设计的应用程序编程接口（API）风格。它遵循一系列的设计原则，以提供可伸缩性、可扩展性和可维护性的 Web 服务。RESTful API 的核心概念是资源（Resources），每个资源都有一个唯一的标识符（URI）进行访问。客户端通过发送 HTTP 请求（GET、POST、PUT、DELETE 等）来操作服务器上的资源。服务器则通过 HTTP 响应返回结果给客户端。

通俗地说，使用了 RESTful API 接口设计规范后，看到 URL 就知道要什么，看到 HTTP method 就知道干什么，看到 HTTP Status code 就知道结果如何。

RESTful API 的特点

  1. 基于标准的HTTP协议：RESTful API使用HTTP协议作为通信协议，利用HTTP的各种方法（GET、POST、PUT、DELETE等）对资源进行操作。
  2. 无状态性（Stateless）：服务器不会存储客户端的任何状态信息，每个请求都是独立的。客户端必须提供完整的请求信息，包括认证和权限验证，以便服务器正确处理请求。
  3. 资源导向（Resource-oriented）：RESTful API将系统中的实体（资源）视为主要的设计元素，每个资源都有一个唯一的URI作为标识符，并通过HTTP方法对其进行操作。
  4. 统一的接口（Uniform Interface）：RESTful API使用统一的接口规范，包括使用标准的HTTP方法来操作资源，使用HTTP状态码来表示操作结果，使用合适的数据格式（如JSON、XML）进行数据交换。
  5. 可缓存性（Cachable）：RESTful API通过使用缓存机制来提高性能和可扩展性。服务器可以使用HTTP头部信息来指示哪些响应可以被缓存，客户端可以使用缓存减少对服务器的请求。
  6. 分层系统（Layered System）：RESTful API支持系统的分层架构，每个层都具有特定的功能和责任。这种分层结构可以提高系统的可扩展性和灵活性。

RESTful API 和传统 API 的对比

RESTful API 更加简洁、灵活，更符合 Web 的原生设计，易于理解和使用。它利用 HTTP 协议和标准的接口规范，提供了一种统一、可伸缩的方式来构建和集成 Web 服务。传统 API 在某些场景下可能仍然有其优势，如强类型、面向对象的编程环境中。选择使用哪种 API 取决于具体的需求和技术栈。RESTful API 和传统 API 在设计和使用方式上存在一些区别，下面是它们之间的对比：

  1. 通信协议：传统 API 通常使用自定义的协议或者 SOAP（Simple Object Access Protocol）作为通信协议，而 RESTful API 使用 HTTP作为通信协议。RESTful API 借助 HTTP 的优势，如广泛支持、缓存机制等。
  2. 接口风格：传统API通常基于RPC（Remote Procedure Call）方式，通过方法调用来进行操作。而RESTful API基于资源导向的设计理念，通过HTTP方法对资源进行操作，更符合Web的原生思维。
  3. 数据交换格式：传统API通常使用XML作为数据交换格式，而RESTful API可以使用多种格式，如JSON、XML、HTML等。JSON已经成为RESTful API中最常用的数据交换格式，因其轻量、易读性好。
  4. 路径设计：传统API的路径通常是基于动词和名词组合（如/api/getUser），而RESTful API的路径应该是基于名词（资源）的层级结构（如/api/users）。
  5. 状态管理：传统API通常在服务端存储状态信息，需要维护会话（Session）状态。而RESTful API是无状态的，每个请求都包含完整的信息，服务器不需要存储任何状态。
  6. 错误处理：传统API通常使用自定义的错误码和错误信息来表示错误情况，而RESTful API使用HTTP状态码来表示不同的操作结果和错误类型。
  7. 安全性：传统API通常使用自定义的安全机制，如基于令牌（Token）的认证。而RESTful API可以使用HTTP的标准认证机制，如基本认证（Basic Authentication）或者OAuth 2.0。

RESTful 设计原则

资源和 URI 的设计

在RESTful架构中，资源是系统中的核心概念，URI（Uniform Resource Identifier）用于唯一标识和定位这些资源。以下是关于资源和URI设计的一些建议：

  1. 使用名词作为URI的路径：URI应该使用名词来表示资源，而不是动词。例如，对于用户资源，可以使用"/users"作为URI。
  2. 使用复数形式：为了保持一致性，建议使用资源的复数形式作为URI的路径，以表示该资源是一个集合。例如，"/users"表示所有用户资源。
  3. 使用层次结构的URI路径：如果资源具有层次结构关系，可以在URI中使用层级路径来表示。例如，"/users/{userId}/posts"表示特定用户的所有帖子资源。
  4. 避免使用动词或操作在URI中：RESTful API的操作应该使用HTTP方法来表示，而不是在URI中使用动词。例如，使用HTTP POST方法在"/users" URI上创建新用户，而不是使用"/createUser"。
  5. 使用URI参数进行过滤和排序：如果需要对资源进行过滤、排序或分页，可以使用URI参数来传递相应的选项。例如，"/users?status=active&sort=name"表示获取状态为活跃且按姓名排序的用户列表。
  6. 保持URI的简洁和可读性：尽量保持URI短小简洁，避免在URI中包含过多的冗余信息。同时，也要考虑URI的可读性，使其易于理解和记忆。

状态码的使用

在RESTful架构中，HTTP状态码是表示请求处理结果的标准化代码。使用适当的状态码可以提供关于请求状态和错误情况的有用信息，帮助客户端和服务器进行适当的处理。下面是一些常见的HTTP状态码及其使用场景：

状态码 含义 解释
200 OK 表示请求成功，并且返回了请求的资源。这是最常见的状态码之一
201 Created 表示请求成功，并且服务器已经创建了新的资源。通常在创建资源的POST请求成功后返回
204 No Content 表示请求成功，但响应中不包含任何内容。通常在删除资源的DELETE请求成功后返回
400 Bad Request 表示请求有语法错误或参数错误，服务器无法理解。客户端应该修改请求后再次发送
401 Unauthorized 表示请求需要身份验证，但客户端未提供有效的身份验证信息。客户端应该提供有效的身份验证凭据后再次发送请求
403 Forbidden 表示服务器理解请求，但拒绝执行。通常表示客户端没有访问请求资源的权限
404 Not Found 表示请求的资源不存在或未找到。客户端请求的资源在服务器上不存在
405 Method Not Allowed 表示请求的HTTP方法不被允许。例如，对于只允许GET请求的资源，如果客户端发送了POST请求，则可以返回405状态码
500 Internal Server Error 表示服务器内部错误，无法完成请求。这通常表示服务器端发生了一些意外错误
503 Service Unavailable 表示服务器当前无法处理请求，通常是由于过载或维护导致的临时性不可用

以上只是一些常见的HTTP状态码，还有其他状态码可供使用。在设计RESTful API时，应根据具体情况选择适当的状态码来准确表示请求结果，并提供有用的信息给客户端。同时，还需遵循HTTP协议的规范和标准使用状态码。

数据格式和内容协商

在 RESTful 架构中，数据格式和内容协商是指客户端和服务器之间根据请求的 Accept 头部字段和服务器的 Content-Type 头部字段来确定传输的数据格式。以下是关于 RESTful 数据格式和内容协商的一些指导原则：

  1. 数据格式：
    • JSON（JavaScript Object Notation）：JSON 是一种轻量级的数据交换格式，易于阅读和编写，并且广泛用于 Web API 的数据传输。
    • XML（eXtensible Markup Language）：XML 是一种标记语言，可用于结构化数据的传输和存储，虽然较为冗长，但具有良好的扩展性。
  2. Accept 头部字段：客户端可以使用 Accept 头部字段来指定它所期望的响应数据格式。例如，Accept: application/json 表示客户端希望接收 JSON 格式的响应数据。
  3. Content-Type 头部字段：服务器使用 Content-Type 头部字段来通知客户端所返回的响应数据格式。例如，Content-Type: application/json 表示服务器返回的数据是 JSON 格式。
  4. 内容协商方式：
    • 媒体类型优先（Media Type Prior）：服务器根据客户端的 Accept 头部字段，选择最适合的媒体类型作为响应数据格式。
    • URI 扩展名（URI Extension）：通过在 URI 中使用文件扩展名来指定数据格式，例如 “/users.json” 表示返回 JSON 格式的用户数据。
    • 查询参数（Query Parameter）：通过在 URI 中使用查询参数来指定数据格式，例如 “/users?format=json” 表示返回 JSON 格式的用户数据。
    • 请求头部字段（Request Header）：客户端将期望的数据格式直接放在请求头部字段中，例如 “X-Accept: application/json” 表示期望返回 JSON 格式的响应数据。
  5. 默认数据格式：如果客户端没有明确指定 Accept 头部字段，则服务器可以选择默认的数据格式作为响应数据的格式。

在设计 RESTful API 时，应该支持常见的数据格式，并允许客户端通过适当的方式指定所需的数据格式。这样可以提高灵活性和互操作性，使得不同类型的客户端可以根据自身需求进行数据交互。同时，还需要在 API 文档中清晰地说明支持的数据格式和内容协商方式，以便客户端开发人员能够正确使用和处理数据。

安全和认证机制

RESTful 接口的安全和认证机制主要包括以下方面：

  1. HTTPS：通过使用 HTTPS 协议来保护传输过程中的数据安全，防止数据被窃听或篡改。
  2. 认证：在 RESTful 接口中，常用的认证方式有基本认证（Basic Authentication）、令牌认证（Token Authentication）等。其中，基本认证是一种最简单的认证方式，需要在请求头中添加 Authorization 字段，并将用户名和密码进行 Base64 编码后传输。令牌认证则需要生成一个令牌，并在每次请求时将令牌放入请求头或参数中进行验证。
  3. 授权：授权是指对用户进行访问控制，确保用户只能访问其具有权限的资源。在 RESTful 接口中，常用的授权方式有基于角色的访问控制（Role-based Access Control）和基于资源的访问控制（Resource-based Access Control）等。
  4. 防止 CSRF：CSRF（Cross-Site Request Forgery）是一种跨站点请求伪造攻击，攻击者通过伪造用户的请求向服务器发送恶意请求。为了防止 CSRF 攻击，可以在每个请求中添加 CSRF Token，并在服务器端进行验证。
  5. 防止 XSS：XSS（Cross-Site Scripting）是一种跨站点脚本攻击，攻击者通过注入恶意脚本代码，来获取用户的敏感信息。为了防止 XSS 攻击，可以对用户输入的内容进行过滤和转义。

在实际开发中，应该根据具体需求和安全风险等级，选择合适的安全和认证机制，并采取适当的措施来加强防护和保护。同时，还需要定期进行安全评估和漏洞扫描，及时修复和升级系统，提高系统的安全性和可靠性。

RESTful API 文档的基本结构

制作 RESTful API 文档时需要考虑到文档的目标受众，以及 API 的具体需求和特点。在编写文档时，需要对 API 进行充分的了解，并根据实际情况进行补充和完善。同时，为了提高文档的可读性和可维护性，还需要遵循一些通用的文档编写规范和标准。

当需要编写一篇 RESTful API 文档时，建议包含如下这些内容：

  1. 概述和背景；
  2. 接口列表和说明；
  3. 接口参数和请求方式；
  4. 请求/响应示例；
  5. 错误处理和状态码说明；
  6. 参考文档和链接。

以下是针对上述的内容进行详细介绍。

概述和背景

编写 RESTful API 文档的概述和背景是为了向读者介绍 API 的基本信息和背景，让他们对该 API 有一个整体的了解。下面是一个编写 RESTful 文档概述和背景的示例：

概述

本文档描述了 [API名称] 的 RESTful API 接口。[API名称] 是一个提供 [功能] 的服务，通过该 API，用户可以实现 [具体功能]。

背景

随着互联网的发展，越来越多的应用程序需要通过接口进行数据交互和业务处理。RESTful API 是一种常用的架构风格，它基于 HTTP 协议，使用统一的资源定位符（URL）来访问和操作数据。相比于传统的 SOAP 等协议，RESTful API 更加简单、灵活、易于理解和使用。

[API名称] 作为一个 [功能描述] 的服务，为了满足不同应用场景的需求，设计了一套符合 RESTful 风格的 API 接口。这些接口通过标准的 HTTP 方法（GET、POST、PUT、DELETE）和 URL 进行访问，使用 JSON 格式进行数据传输，提供了灵活且易于集成的方式，方便开发者快速构建应用并与服务进行交互。

本文档将详细介绍每个接口的使用方法、参数说明、返回值格式和可能出现的错误码。通过阅读本文档，开发者可以快速理解和使用 [API名称] 的 RESTful 接口，实现对 [功能] 的集成和调用。

以上是一个编写 RESTful API 文档概述和背景的示例，根据实际情况，你可以根据项目的特点和需求进行适当的修改和补充。重点是清晰地介绍 API 的概述、作用以及为什么选择使用 RESTful 架构风格。

接口参数和请求方式

请求方式（HTTP 方法）：

  • GET：用于获取资源，不会对服务器产生副作用。
  • POST：用于创建新资源或提交数据，可能会对服务器产生副作用。
  • PUT：用于更新已存在的资源，要求提供完整的资源信息。
  • PATCH：用于更新已存在的资源，只需要提供需要更新的字段。
  • DELETE：用于删除指定的资源。

参数传递方式：

  • 查询参数（Query Parameters）：通过 URL 的查询字符串传递参数，一般用于 GET 请求。例如：/users?name=John&age=25
  • 路径参数（Path Parameters）：将参数嵌入到 URL 的路径中，用于标识资源的唯一性。例如：/users/1
  • 请求体（Request Body）：将参数包含在请求体中，通常用于 POST、PUT 和 PATCH 请求，可以使用 JSON、XML 等格式进行数据传输。

参数类型：

  • 必填参数：必须提供的参数，请求无法正常执行或返回错误。
  • 可选参数：不是必须提供的参数，根据业务需求来决定是否提供。

请求/响应示例

获取用户信息接口

  • 请求方式：GET

在 RESTful API 中，错误处理和状态码是非常重要的组成部分。对于错误处理，可以返回一个包含错误信息的 JSON 对象，例如：

jsonCopy Code{
  "error": "Invalid input",
  "message": "The request body is missing required fields."

或者使用自定义的错误代码和描述，例如：

jsonCopy Code{

RESTful API文档工具和平台

Swagger和OpenAPI规范

Swagger和OpenAPI规范是用于描述和定义RESTful API的工具和规范。

  • Swagger最初是一个开源项目，它提供了一种简洁、易读的方式来描述API的各种信息，包括请求和响应的参数、数据类型、路径、HTTP方法等。Swagger可以生成可交互的API文档，使开发者能够轻松理解和使用API。

  • OpenAPI规范则是Swagger规范的演化版本，它由OpenAPI Initiative（OAI）维护。OpenAPI规范基于JSON或YAML格式，提供了一种标准化的方式来描述和定义API的结构、操作和参数。它不仅可以生成API文档，还可以用于生成客户端代码、服务器存根、数据验证等。

通过Swagger和OpenAPI规范，开发者能够更好地理解和使用API，同时也能够促进API的交互性和可重用性。许多开发工具和框架都支持Swagger和OpenAPI规范，使得API的开发和集成更加方便和高效。

Postman和Insomnia测试工具

Postman和Insomnia都是常用的API测试工具。

  • Postman是一个功能强大、易于使用的API测试和开发工具。它提供了一个可视化界面，使得开发者能够快速、轻松地创建和发送HTTP请求，并查看响应结果。Postman还支持请求集合的管理、自动化测试、环境变量等功能，可以大大提高开发效率。

  • Insomnia也是一个跨平台的API测试和开发工具。它提供了一个清晰、直观的用户界面，支持多种HTTP方法、请求参数和响应格式。Insomnia还支持请求和响应的截图、代码生成、环境变量等功能，可以帮助开发者更好地理解和测试API。

两者虽然功能类似，但在使用中还是有一些区别。例如，Postman提供了在线API文档、协作和团队管理等功能，在多人协作的场景下更加方便。而Insomnia则更注重界面的美观和易用性。

Gitbook、Readme和Confluence文档平台

GitBook、README和Confluence都是常用的文档平台，用于编写和分享技术文档。

  • GitBook是一个开源的文档平台，它使用Markdown格式来编写文档，并提供了强大的版本控制和协作功能。通过GitBook，你可以将文档存储在Git仓库中，并与团队成员共享和协作编辑。GitBook还支持自动生成漂亮的HTML或PDF格式的文档，方便阅读和分享。

  • README是一种简单的文档格式，通常用于描述项目的基本信息和使用方法。README文件通常以纯文本形式存在于项目的根目录下，它可以包含项目的介绍、安装指南、使用示例等内容。README文件通常直接存储在代码仓库中，与代码一同被版本控制管理。

  • Confluence是一个企业级的协作文档平台，它提供了丰富的文档编辑和组织功能。Confluence支持团队成员协作编辑、讨论、评论等，可以方便地创建和维护复杂的技术文档。Confluence还提供了强大的搜索和权限控制功能，适用于大型团队和企业内部的知识管理和文档协作。

根据具体的需求和使用场景，选择合适的文档平台很重要。GitBook适合开源项目或小团队，强调版本控制和协作编辑；README适用于简单的项目文档，方便与代码一同管理；Confluence则适合大型团队和企业，提供全面的协作和管理功能。
